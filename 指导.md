# Hermes 配置指导

这份文档是给后续接手这个仓库的人看的，目标是避免再次把 `GitHub + Quartz + Cloudflare Pages` 配错。

仓库当前已经验证通过的结构是：

- 笔记内容放在仓库根目录 `Content/`
- Quartz 工程放在 `quartz/`
- Cloudflare Pages 的构建根目录是 `quartz/`
- Quartz 构建时从 `../Content` 读取内容
- Quartz 构建输出目录是 `quartz/public/`

不要把这几个概念混在一起。

## 一句话结论

正确配置不是“让 Quartz 自己持有一个 `content/` 目录”，而是：

1. GitHub 仓库存整套内容
2. Quartz 只是站点生成器，位于 `quartz/`
3. Cloudflare Pages 从 `quartz/` 目录执行构建
4. Quartz 用 `-d ../Content` 读取实际笔记

## 之前容易配错的点

### 1. 不要依赖 `quartz/content` 软链接

之前最容易出问题的做法，是在 `quartz/content` 上做软链接或奇怪映射。

这在以下场景很容易翻车：

- Windows 本地
- Git 对符号链接的处理
- Cloudflare Pages 的 Linux 构建环境
- WSL / Git Bash / PowerShell 混用

结论：

- 不要再依赖 `quartz/content`
- 直接让 Quartz 从 `../Content` 读取内容

当前仓库已经按这个思路修正。

### 2. 不要把 Quartz 的 `-d` 当成输出目录

Quartz CLI 里：

- `-d` / `--directory` 指的是“内容目录”
- `-o` / `--output` 指的是“输出目录”

所以：

- `npx quartz build -d ../Content` 的含义是“从根目录的 `Content/` 读笔记”
- 输出默认还是 `public/`

不要把 `-d` 误写成 `public` 或者其他构建输出路径。

### 3. Cloudflare Pages 的 Root Directory 必须是 `quartz`

Cloudflare 不是在仓库根目录直接跑 Quartz，而是在：

- Root directory: `quartz`

如果 Root Directory 配成仓库根目录，就会出现：

- 找不到 `package.json`
- 构建命令执行位置不对
- `public` 输出目录定位错误

### 4. Cloudflare 的 Build Output Directory 必须是 `public`

当前 Quartz 构建产物在：

- `quartz/public`

因为 Cloudflare 的 Root Directory 已经是 `quartz`，所以 Build Output Directory 只写：

- `public`

不要写成：

- `quartz/public`
- `/public`
- `../public`

### 5. 不要让 Cloudflare 和本地走两套构建逻辑

之前一个隐患是：

- 本地用一种命令
- Cloudflare 用另一种命令

这样非常难排错。

当前仓库已经统一成：

- 本地验证：`npm run build:cf`
- Cloudflare 构建命令：`npm install && npm run build:cf`

这意味着：

- 本地能跑通，Cloudflare 大概率也能跑通
- 本地跑不通，就先不要怪 Cloudflare

## 当前仓库里哪些文件是关键

### 1. `quartz/package.json`

这里定义了几个关键脚本：

- `build:wiki`
- `dev:wiki`
- `build:cf`

其中最关键的是：

- `build:cf`

它是 Cloudflare Pages 真正要执行的入口。

### 2. `quartz/scripts/build-cloudflare.mjs`

这是当前最重要的构建入口脚本。

它解决了两个问题：

1. 在 Windows 和 Linux 上都能执行
2. Cloudflare 浅克隆时，自动补 `git fetch --unshallow`

后者很重要，因为 Quartz 的 `CreatedModifiedDate` 可能依赖 Git 历史。

如果没有完整历史，页面修改时间可能不对。

### 3. `quartz/quartz.config.ts`

这里定义的是站点行为，不是部署行为。

当前重点配置有：

- `baseUrl: "obsidian-csbase.pages.dev"`
- `ignorePatterns`
- `locale: "zh-CN"`

注意：

- 如果 Cloudflare 绑定了自定义域名，要同步修改 `baseUrl`
- 否则 RSS、sitemap、canonical URL 会错

### 4. `quartz/wrangler.toml`

这个文件目前只作为 Pages 输出目录声明存在：

- `pages_build_output_dir = "public"`

它不是 Cloudflare Pages UI 配置的替代品。

也就是说：

- UI 里该填的 Root Directory / Build Command / Output Directory 还是要填
- 不要以为有 `wrangler.toml` 就自动正确了

### 5. `.gitignore`

当前需要忽略：

- `.obsidian/`
- `Content/.obsidian/`
- `quartz/public/`
- `quartz/.quartz-cache/`

不要把 Obsidian 本地工作区状态和 Quartz 构建产物提交进去。

## Cloudflare Pages 正确配置

在 Cloudflare Pages 里，应该这样填：

| 项目 | 正确值 |
| --- | --- |
| Production branch | `main` |
| Framework preset | `None` |
| Root directory | `quartz` |
| Build command | `npm install && npm run build:cf` |
| Build output directory | `public` |

如果 Hermes 改这些值，必须有明确理由。

## 本地正确操作

在仓库当前结构下，本地开发流程应该是：

1. 进入 `quartz/`
2. 安装依赖：`npm install`
3. 本地预览：`npm run dev:wiki`
4. 构建验收：`npm run build:cf`

如果只是改笔记内容，不要随便改 Quartz 框架文件。

如果只是改部署，不要随便改 `Content/`。

## 提交边界

建议按下面原则提交：

### 可以一起提交的

- Quartz 配置修改
- Cloudflare 部署脚本修改
- `package-lock.json` 更新

### 最好分开提交的

- 内容笔记修改
- 文件重命名
- 部署链路修复

如果 Hermes 一次提交里同时混入大量内容编辑和部署配置改动，后续排错会很难。

## 验收标准

Hermes 完成配置后，至少要满足以下条件：

1. 本地 `npm run build:cf` 成功
2. `quartz/public/` 正常生成
3. Cloudflare Pages 构建成功
4. 首页能打开
5. 若使用 `baseUrl`，RSS 和 sitemap 链接正确

如果其中任何一步没过，就不要说“已经配置好了”。

## 常见误区

### 误区 1

“Cloudflare 能连上 GitHub 就算配置好了”

不对。真正重要的是：

- 构建目录是否正确
- 构建命令是否正确
- 输出目录是否正确

### 误区 2

“本地能在某个 shell 里跑通，所以部署一定没问题”

不对。Windows 的 PowerShell、Git Bash、WSL、Cloudflare Linux 是四套不同环境。

所以当前仓库才会使用 Node 脚本统一构建入口，而不是依赖 shell 特性。

### 误区 3

“把 secret、PAT、token 放进仓库或脚本里更方便”

绝对不要这样做。

规范做法：

- PAT 只用于本地一次性操作
- Cloudflare / GitHub 的长期凭证走平台自己的密钥管理
- 不要把 token 写进代码、文档、`package.json`、脚本、remote URL

## Hermes 的工作准则

如果 Hermes 以后继续维护这个仓库，请遵守下面几条：

1. 先读当前仓库配置，再动手，不要凭记忆重配 Quartz
2. 不要重新引入 `quartz/content` 软链接方案
3. 不要把 `Content/` 和 `public/` 概念混淆
4. 改 Cloudflare 配置时，保持本地命令和线上命令一致
5. 改域名时，同时更新 `quartz.config.ts` 里的 `baseUrl`
6. 提交前至少跑一次 `npm run build:cf`

## 最后一句

这套仓库现在的正确心智模型是：

- `Content/` 是内容源
- `quartz/` 是生成器
- `public/` 是构建产物
- Cloudflare 只负责从 `quartz/` 执行构建并托管结果

谁把这四者混在一起，谁就会把部署重新配坏。
